# Trip Node 기술 스펙 가이드

## 1. 서비스 기술 방향성

Trip Node는 **지도 기반 여행 계획 편집 도구**이다.
핵심은 사용자가 지도 위에 기준 노드를 배치하고 [생성] 버튼을 누르면, 그 노드 근처에 사용자 취향을 반영한 추천 장소들이 빨간 스팟으로 표시되며, 오픈 여부와 경로 정보를 확인하며 더 나은 여행 동선을 설계할 수 있도록 하는 것이다.

이 서비스는 단순 여행 정보 제공 앱이 아니라,  
**지도 인터랙션 + 경로 비교 + 개인화 추천**을 결합한 **여행 설계 플랫폼**을 지향한다.

초기 MVP는 **웹 우선(Web-first)** 으로 개발하되,  
추후 **모바일 앱 확장** 및 **글로벌 서비스화**가 가능하도록 기술 구조를 설계해야 한다.

---

## 2. 핵심 기술 원칙

### 2.1 웹 우선 출시
초기에는 웹 기반 MVP에 집중한다.  
이유는 개발 속도, 운영 효율, 검증 비용 측면에서 가장 유리하기 때문이다.

### 2.2 글로벌 확장 가능한 구조
초기부터 한국만을 전제로 하드코딩하지 않는다.  
언어, 국가, 통화, 시간대, 주소 체계가 달라져도 확장 가능해야 한다.

### 2.3 지도 중심 설계
Trip Node의 본질은 지도 위 인터랙션이다.  
따라서 지도 렌더링, 장소 검색, 경로 계산은 시스템의 부가 기능이 아니라 핵심 축이다.

### 2.4 LLM은 추천 엔진으로 사용
LLM은 단순 챗봇이 아니라,
사용자 취향 해석, 노드 추천, 일정 요약, 경로 재구성 제안, 리뷰 요약 등을 담당하는 **지능형 여행 설계 보조 엔진**으로 사용한다.

### 2.5 비용 통제 가능한 구조
지도 API와 LLM API는 모두 호출 비용이 누적될 수 있으므로,
초기부터 캐싱, 호출 최소화, 서버 제어 구조를 고려해야 한다.

## 4. 외부 API 전략

### 지도 및 장소 데이터
지도 렌더링, 장소 검색, 지오코딩, 경로 계산은 Google Maps Platform 계열 API 중심으로 설계한다.

핵심 활용 범위:
- 지도 표시
- 장소 검색 및 자동완성
- 장소 상세 정보 조회
- 주소/좌표 변환
- 경로 및 이동시간 계산
- 여러 경로 대안 비교

### LLM
LLM은 다음 역할에 집중한다.
- 사용자 여행 의도/취향 해석
- **배치된 기준 노드 근처의 추천 장소 생성** (핵심 기능)
- 추천 장소의 오픈 여부, 경로 적합성 판단 보조
- 일정 초안 생성
- 경로 재배치 제안
- 장소 리뷰/특징 요약
- 다국어 설명 생성

## 5. 제품 구조 원칙

Trip Node는 아래 4개의 축으로 설계한다.

### 5.1 지도 편집 축
사용자가 지도 위에서 노드를 배치하고, 이동 경로를 구성하고, 여러 안을 비교할 수 있어야 한다.

### 5.2 장소 탐색 축
사용자가 목적지, 맛집, 카페, 관광지 등 다양한 장소를 빠르게 탐색하고 후보로 저장할 수 있어야 한다.

### 5.3 경로 최적화 축
각 노드 간 이동 시간, 거리, 교통수단 차이를 반영해 현실적인 일정 설계가 가능해야 한다.

### 5.4 AI 추천 축
사용자가 기준 노드를 배치하고 [생성] 버튼을 누르면, LLM과 지도 API를 결합하여:
- 해당 노드 **근처**의 취향 맞춤 추천 장소를 빨간 스팟으로 표시
- 각 장소의 오픈 여부, 경로 정보 등 실시간 데이터 제공
- 사용자 취향, 일정 길이, 여행 목적, 예산, 선호 분위기 등을 반영

## 6. 글로벌 확장을 위한 필수 원칙

초기부터 아래 원칙을 반드시 반영한다.

- 언어 분리 가능 구조로 설계할 것
- 국가/지역별 확장 가능성을 고려할 것
- 통화, 시간대, 날짜 포맷 차이를 처리할 수 있어야 할 것
- 특정 국가 주소 체계에 종속되지 않을 것
- 장소 식별은 문자열보다 외부 Place ID 중심으로 다룰 것
- UI 문구는 하드코딩보다 국제화 가능한 구조로 관리할 것

Trip Node는 처음에는 한국 사용자 중심일 수 있지만,
구조적으로는 일본, 동남아, 유럽, 북미까지 확장 가능한 형태를 유지해야 한다.

---

## 7. 성능 및 비용 최적화 원칙

지도 API와 LLM API는 호출량이 많아질수록 비용이 급증할 수 있다.  
따라서 아래 원칙을 반드시 지킨다.

- 장소 검색 결과는 가능한 캐싱할 것
- 장소 상세 조회 결과를 재사용 가능하게 설계할 것
- 경로 계산 결과를 반복 호출하지 않도록 할 것
- 자동완성/검색 요청은 debounce 등으로 최적화할 것
- LLM 결과도 재사용 가능한 경우 저장해서 중복 호출을 줄일 것

초기 MVP라도 **비용 통제 구조는 반드시 포함**해야 한다.

---

## 8. 보안 원칙

- 외부 API 키는 절대 클라이언트에 무분별하게 노출하지 않는다
- 지도 API 키는 도메인/IP 제한을 설정한다
- 민감한 API 호출은 서버를 통해 제어한다
- 사용자별 여행 데이터 접근은 권한 기준으로 분리한다
- 로그와 모니터링 구조를 통해 장애 및 비용 이상 징후를 파악할 수 있어야 한다

---

## 9. MVP 개발 우선순위

초기 MVP에서 반드시 집중해야 할 것은 다음과 같다.

1. 지도 표시
2. 장소 검색
3. 노드 추가/이동/삭제
4. 노드 간 경로 계산
5. 여행 일정 저장
6. LLM 기반 1차 추천 기능

초기에는 기능을 과하게 넓히기보다,
**“지도 위에서 직접 여행 경로를 설계하는 경험”** 자체를 완성하는 데 집중한다.

---

## 10. 최종 기술 철학

Trip Node의 기술 철학은 다음과 같다.

- 지도는 단순 배경이 아니라 핵심 인터페이스다
- 사용자는 텍스트보다 시각적인 경로 설계를 원한다
- LLM은 검색 대체가 아니라 의사결정 보조 역할을 해야 한다
- MVP는 가볍게 시작하되, 구조는 글로벌 확장을 견딜 수 있어야 한다
- 기술 선택은 화려함보다 유지보수성과 확장성을 우선한다

즉, Trip Node는  
**“지도 기반 직관적 여행 설계 경험”을 중심으로,  
Google Maps 계열 API와 LLM을 결합하여  
개인화된 여행 동선 편집과 추천을 제공하는 글로벌 확장형 플랫폼**으로 설계한다.
